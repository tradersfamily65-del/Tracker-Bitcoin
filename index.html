Abstrak
Ringkasan Dokumen
"Versi murni dari uang elektronik peer-to-peer akan memungkinkan pembayaran online dikirim langsung dari satu pihak ke pihak lain tanpa melalui lembaga keuangan."
Dokumen ini memaparkan solusi untuk masalah double-spending (pembelanjaan ganda) dalam sistem digital menggunakan jaringan peer-to-peer. Jaringan ini memberi timestamp pada transaksi dengan cara memasukkannya ke dalam rantai hash Proof-of-Work yang berkelanjutan — membentuk catatan yang tidak dapat diubah tanpa mengulangi seluruh pekerjaan komputasi.
Bagian 1
Pendahuluan
Perdagangan di internet hampir sepenuhnya bergantung pada lembaga keuangan yang bertindak sebagai pihak ketiga yang terpercaya untuk memproses pembayaran elektronik. Meskipun sistem ini bekerja cukup baik untuk sebagian besar transaksi, sistem ini masih memiliki kelemahan bawaan dari model berbasis kepercayaan.

Transaksi yang tidak dapat dibatalkan tidak benar-benar mungkin dilakukan karena lembaga keuangan tidak dapat menghindari mediasi sengketa. Biaya mediasi meningkatkan biaya transaksi, membatasi ukuran transaksi minimum yang praktis, dan memutus kemungkinan transaksi kecil yang kasual.

Yang dibutuhkan adalah sistem pembayaran elektronik berbasis bukti kriptografis, bukan kepercayaan — memungkinkan dua pihak manapun untuk bertransaksi langsung satu sama lain tanpa membutuhkan pihak ketiga yang terpercaya.
SISTEM LAMA (Bank): Pengirim → Bank (pihak ketiga) → Penerima ↑ biaya, lambat, bisa ditolak SISTEM BITCOIN: Pengirim ─────────────────→ Penerima peer-to-peer langsung
Bagian 2
Transaksi
Kami mendefinisikan koin elektronik sebagai rantai tanda tangan digital. Setiap pemilik mentransfer koin kepada orang berikutnya dengan menandatangani secara digital sebuah hash dari transaksi sebelumnya dan kunci publik pemilik berikutnya, lalu menambahkannya ke ujung koin.

Masalahnya adalah penerima pembayaran tidak dapat memverifikasi bahwa salah satu pemilik tidak membelanjakan koin yang sama dua kali. Solusi umum adalah memperkenalkan otoritas pusat yang terpercaya — tapi ini menciptakan titik kegagalan tunggal.
Transaksi 1 Transaksi 2 Transaksi 3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Kunci │ │ Kunci │ │ Kunci │ │ Publik │ → │ Publik │ → │ Publik │ │ Pemilik2 │ │ Pemilik3 │ │ Pemilik4 │ ├──────────┤ ├──────────┤ ├──────────┤ │ Hash Tx │ │ Hash Tx │ │ Hash Tx │ │ Sebelum │ │ Sebelum │ │ Sebelum │ ├──────────┤ ├──────────┤ ├──────────┤ │ Tanda │ │ Tanda │ │ Tanda │ │ Tangan │ │ Tangan │ │ Tangan │ │ Pemilik1 │ │ Pemilik2 │ │ Pemilik3 │ └──────────┘ └──────────┘ └──────────┘
Solusi Bitcoin: semua transaksi diumumkan secara publik dan semua peserta sepakat pada satu riwayat urutan transaksi yang diterima.
Bagian 3
Server Timestamp (Cap Waktu)
Solusinya dimulai dengan server timestamp. Server timestamp bekerja dengan mengambil hash dari sekumpulan item yang perlu diberi timestamp dan mempublikasikan hash tersebut secara luas.

Timestamp membuktikan bahwa data pasti sudah ada pada waktu tersebut. Setiap timestamp menyertakan timestamp sebelumnya dalam hash-nya, membentuk rantai — setiap timestamp memperkuat yang sebelumnya.
Block 1 Block 2 Block 3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Hash │ ───→ │ Hash │ ───→ │ Hash │ │ Block 0 │ │ Block 1 │ │ Block 2 │ ├──────────┤ ├──────────┤ ├──────────┤ │ Data │ │ Data │ │ Data │ │Transaksi │ │Transaksi │ │Transaksi │ └──────────┘ └──────────┘ └──────────┘ BLOCKCHAIN = RANTAI BLOK TERKONEKSI
Bagian 4
Proof-of-Work (Bukti Kerja)
Untuk mengimplementasikan timestamp terdistribusi peer-to-peer, kita perlu menggunakan sistem Proof-of-Work seperti Hashcash oleh Adam Back.

Proof-of-Work melibatkan pemindaian nilai tertentu (nonce) yang ketika di-hash — misalnya dengan SHA-256 — hash-nya dimulai dengan sejumlah bit nol. Pekerjaan rata-rata yang diperlukan adalah eksponensial dalam jumlah bit nol yang diperlukan.
Target Hash: 0000000000000abc123... (harus dimulai dengan banyak angka nol) Komputer mencoba miliaran kombinasi: Nonce 1: hash = f4a9b2c... ✗ (tidak valid) Nonce 2: hash = 3d8e1f2... ✗ (tidak valid) Nonce X: hash = 0000000... ✓ (VALID! Blok ditemukan) → Pemenang mendapat hadiah Bitcoin baru
Setelah CPU dihabiskan untuk memenuhi Proof-of-Work, blok tidak dapat diubah tanpa mengulangi pekerjaan. Karena blok-blok berikutnya dirantai setelahnya, pekerjaan untuk mengubah blok mencakup mengulang semua blok setelahnya.

Satu suara adalah satu CPU — bukan satu IP address. Mayoritas jujur mengontrol CPU terbanyak dan rantai jujur akan tumbuh paling cepat.
Bagian 5
Jaringan
Langkah-langkah menjalankan jaringan:

1. Transaksi baru disiarkan ke semua node (simpul jaringan).
2. Setiap node mengumpulkan transaksi baru ke dalam sebuah blok.
3. Setiap node bekerja mencari Proof-of-Work yang sulit untuk bloknya.
4. Ketika sebuah node menemukan Proof-of-Work, ia menyiarkan blok tersebut ke semua node.
5. Node menerima blok hanya jika semua transaksi di dalamnya valid dan belum dibelanjakan.
6. Node mengekspresikan penerimaan blok dengan bekerja untuk membuat blok berikutnya dalam rantai.
Node A ←→ Node B ←→ Node C ↕ ↕ ↕ Node D ←→ Node E ←→ Node F ↕ ↕ ↕ Node G ←→ Node H ←→ Node I Ribuan node di seluruh dunia Tidak ada yang bisa matikan semuanya
Node selalu menganggap rantai terpanjang sebagai yang benar dan akan terus bekerja untuk memperluasnya.
Bagian 6
Insentif (Mining Reward)
Secara konvensi, transaksi pertama dalam sebuah blok adalah transaksi khusus yang memulai koin baru yang dimiliki oleh pembuat blok. Ini menambahkan insentif bagi node untuk mendukung jaringan dan menyediakan cara awal mendistribusikan koin ke peredaran.
Reward Awal (2009): 50 BTC per blok Halving 1 (2012): 25 BTC per blok Halving 2 (2016): 12.5 BTC per blok Halving 3 (2020): 6.25 BTC per blok Halving 4 (2024): 3.125 BTC per blok ← Sekarang Halving 5 (~2028): 1.5625 BTC per blok Total: ~21 juta BTC, habis ~tahun 2140
Insentif juga dapat didanai dengan biaya transaksi. Jika output suatu transaksi lebih kecil dari inputnya, selisihnya adalah biaya transaksi yang ditambahkan ke nilai insentif blok tersebut.

"Sekali sejumlah koin yang ditentukan telah beredar, insentif dapat beralih sepenuhnya ke biaya transaksi dan sepenuhnya bebas dari inflasi." — Satoshi Nakamoto
Insentif ini mendorong node untuk tetap jujur. Jika penyerang serakah memiliki lebih banyak CPU dari semua node jujur, ia harus memilih antara menggunakannya untuk menipu orang atau menghasilkan koin baru. Ia seharusnya menemukan lebih menguntungkan untuk bermain sesuai aturan.
Bagian 7
Mempersingkat Ruang Disk
Setelah transaksi terbaru dalam sebuah koin terkubur di bawah blok-blok yang cukup, transaksi yang dibelanjakan sebelumnya dapat dibuang untuk menghemat ruang disk.

Untuk memfasilitasi ini tanpa merusak hash blok, transaksi di-hash dalam Pohon Merkle (Merkle Tree) — hanya root yang disertakan dalam hash blok. Blok lama dapat dipadatkan dengan memangkas cabang-cabang pohon.
Root Hash / \ Hash 01 Hash 23 / \ / \ Hash0 Hash1 Hash2 Hash3 | | | | Tx0 Tx1 Tx2 Tx3 Tx0, Tx1, Tx2 sudah lama → bisa dipangkas Tapi Root Hash tetap valid untuk verifikasi
Bagian 8
Verifikasi Pembayaran Sederhana
Dimungkinkan untuk memverifikasi pembayaran tanpa menjalankan node jaringan penuh. Pengguna hanya perlu menyimpan salinan header blok dari rantai Proof-of-Work terpanjang.

Ini adalah dasar dari apa yang sekarang kita sebut SPV (Simplified Payment Verification) — memungkinkan dompet mobile seperti yang ada di HP-mu untuk memverifikasi transaksi tanpa mengunduh seluruh blockchain (yang sudah ratusan GB).
Bagian 9
Menggabungkan & Memisahkan Nilai
Meskipun akan mungkin untuk menangani koin secara individual, akan tidak praktis untuk melakukan transaksi terpisah untuk setiap sen dalam transfer.

Untuk memungkinkan nilai digabungkan dan dipecah, transaksi berisi beberapa input dan output. Biasanya ada satu input dari transaksi besar sebelumnya atau beberapa input yang menggabungkan jumlah lebih kecil — dan paling banyak dua output: satu untuk pembayaran dan satu mengembalikan kembalian ke pengirim.
INPUT: 0.5 BTC + 0.3 BTC + 0.2 BTC = 1.0 BTC OUTPUT: 0.7 BTC (ke penerima) + 0.29 BTC (kembalian) + 0.01 BTC (biaya transaksi untuk miner)
Bagian 10
Privasi
Model perbankan tradisional mencapai privasi dengan membatasi akses informasi kepada pihak-pihak yang terlibat dan pihak ketiga terpercaya. Bitcoin menempuh pendekatan berbeda.

Semua transaksi harus diumumkan secara publik — tetapi privasi tetap dapat dijaga dengan memutus aliran informasi di tempat lain: dengan menjaga kunci publik tetap anonim.
BANK TRADISIONAL: Identitas ─── Transaksi (rahasia, hanya bank tahu) BITCOIN: Kunci Publik ─── Transaksi (publik, semua bisa lihat) ↑ Tidak ada yang tahu kunci publik ini milik siapa → Privasi terjaga meski transaksi transparan
Sebagai firewall tambahan, sepasang kunci baru harus digunakan untuk setiap transaksi untuk mencegah mereka terhubung ke pemilik yang sama — prinsip yang masih berlaku dalam penggunaan Bitcoin modern.
Bagian 11
Kalkulasi Keamanan
Satoshi memaparkan perhitungan matematis untuk membuktikan bahwa penyerang tidak dapat membalikkan transaksi kecuali ia memiliki lebih dari 50% dari total hash power jaringan.

Bahkan dengan 10% hash power, probabilitas berhasil menyusul rantai jujur turun secara eksponensial seiring bertambahnya kedalaman blok.
P = probabilitas penyerang sukses q = fraksi hash power penyerang z = jumlah blok yang harus dikejar P = 1 − Σ(k=0 to z) [e^(-λ) × λ^k / k!] × (1−(q/p)^(z-k)) Jika q = 0.1 (10% hash power): z=5 blok: P = 0.0010 (0.1%) z=10 blok: P = 0.0001 (0.01%) z=15 blok: P → ~0 (hampir mustahil)
Inilah mengapa standar 6 konfirmasi blok dianggap aman secara kriptografis untuk transaksi besar.
Bagian 12
Kesimpulan
"Kami telah mengusulkan sebuah sistem untuk transaksi elektronik tanpa bergantung pada kepercayaan. Kami mulai dengan kerangka biasa koin yang dibuat dari tanda tangan digital, yang memberikan kendali kepemilikan yang kuat, tetapi tidak lengkap tanpa cara untuk mencegah pengeluaran ganda."
Untuk menyelesaikan ini, Satoshi mengusulkan jaringan peer-to-peer menggunakan Proof-of-Work untuk merekam riwayat publik dari transaksi yang dengan cepat menjadi tidak praktis secara komputasi bagi penyerang untuk diubah jika node-node jujur mengontrol mayoritas CPU.

Jaringan ini kuat dalam kesederhanaan yang tidak terstruktur. Node dapat meninggalkan dan bergabung kembali dengan jaringan sesuka hati, menerima rantai Proof-of-Work terpanjang sebagai bukti dari apa yang terjadi saat mereka pergi.

Mereka memberikan suara dengan CPU mereka, mengekspresikan penerimaan mereka atas blok yang valid dengan mengerjakan blok berikutnya dan menolak blok yang tidak valid dengan menolak untuk mengerjakannya.

Aturan dan insentif apapun dapat diberlakukan dengan mekanisme konsensus ini.
